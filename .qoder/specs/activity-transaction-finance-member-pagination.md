# 活动/事务/财务/社员页面 - 多条内容展示机制分析

## 当前状态

四个页面均采用**一次性加载、一次性渲染**，无分页或"加载更多"机制。

| 页面 | 展示形式 | 后端限制 | 前端分页 | 风险 |
|------|---------|---------|---------|------|
| 活动 | 卡片 | 默认50条 | 无 | 超50条截断 |
| 事务 | 卡片 | get_all全部 | 无 | 内存溢出 |
| 财务 | 表格 | 固定100条 | 无 | 超100条截断+统计不准 |
| 社员 | 网格 | get_all全部 | 无 | 内存溢出 |

## 后端现状

- `JsonlDB.fetch_page()` 已实现完善分页（page/limit/search/reverse）
- 事务和社员API已预留分页入口（page>0 && limit>0 时返回 {data, total, page, limit}）
- 活动API支持page/limit参数但前端未使用
- 财务API硬编码 page=1, limit=100

## 参考实现：藏诗阁（已有分页）

藏诗阁是唯一实现了"加载更多"机制的页面：
- 全局状态：`_poemPage`, `_poemHasMore`, `_cachedPoems`
- 每次加载10条，判断返回数<limit则无更多
- "加载更多"按钮追加渲染
- 支持搜索重置分页
